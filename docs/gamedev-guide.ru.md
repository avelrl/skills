# Gamedev Guide

Понятное руководство по `gamedev`-процессу, чтобы не приходилось сначала разбирать все внутренние документы и навыки.

Связанные документы:

- быстрый старт: `docs/gamedev-quickstart.ru.md`
- разбор типовых проблем: `docs/gamedev-troubleshooting.ru.md`
- глоссарий: `docs/gamedev-glossary.ru.md`
- канонический сценарий: `docs/gamedev-workflow.md`

## Что зафиксировать в самом начале

До любого серьезного запроса к Codex лучше явно зафиксировать четыре вещи:

1. идея или фантазия игры
2. текущий этап: концепция, MVP или демо-версия
3. рабочий язык документации
4. язык или языки для игрока

Разделяй эти вещи явно:

- `Working / Documentation Language`: язык GDD, отчетов и проектной документации
- `Player-Facing Language(s)`: язык меню, HUD, промптов и игрового текста

Не стоит надеяться, что агент сам это правильно выведет из общего стиля переписки.
Лучше сказать это в первом же запросе на создание концепции.

## Если есть только концепция и нужен первый GDD

Это место, где чаще всего возникает путаница.
`design-system` обычно не должен стартовать прямо из одной только концепции.

Правильный путь такой:

1. есть `design/gdd/game-concept.md`
2. запускается `/setup-engine`
3. потом `/map-systems`
4. только после этого вызывается `/design-system [system-name]`

Причина простая:

- `design-system` опирается на `design/gdd/systems-index.md`
- `systems-index` появляется на шаге `map-systems`
- без этой карты легко начать писать GDD для системы, которая еще не определена или не приоритетна

Если хочешь получить первый GDD как можно раньше, лучше дать такой запрос:

```text
У меня есть только `design/gdd/game-concept.md`.
Не переходи сразу к `/design-system`.
Сначала используй `/setup-engine`, потом `/map-systems`.
После этого предложи, для какой системы первым вызвать `/design-system`.
Рабочий язык документации: русский.
Язык для игрока: английский.
```

## Основной путь: от идеи к MVP

Используй этот путь, когда тебе нужно сначала доказать основной игровой цикл.

1. `setup-engine`
   Результат: `docs/technical-preferences.md`
2. `map-systems`
   Результат: `design/gdd/systems-index.md`
3. `design-system`
   Результат: один GDD для системы, например `design/gdd/combat-loop.md`
4. `prototype`, только если еще есть один реальный риск, который мешает уверенно зафиксировать дизайн
5. `bootstrap-project`
   Результат: минимальный рабочий каркас проекта
6. `implement-system`
   Результат: одна утвержденная система в рабочем коде
7. `assemble-mvp`
   Результат: `reports/mvp-assembly-report.md`
8. `playtest-and-tune`
   Результат: `reports/playtest-report.md`

## Дополнительный путь: от MVP к демо

Используй этот путь, когда игровой цикл уже работает, а следующая цель не просто доказать MVP, а показать внятную демо-версию другим людям.

1. `prepare-demo`
   Результат: `reports/demo-readiness.md`
2. `design-system` или `implement-system` для самого важного препятствия к демо
3. снова `assemble-mvp`, чтобы подтвердить, что обновленный срез по-прежнему работает как целое
4. снова `playtest-and-tune`, чтобы проверить уже уровень качества, нужный для демо

`prepare-demo` не только про UI.
Это шаг, где фиксируется контракт демо-версии для всего среза:

- читаемость игрового процесса
- сопровождение игрока и первые подсказки
- HUD, меню и подсказки
- проигрыш, перезапуск и выход
- правила по временным заглушкам
- нехватка контента и ассетов
- ожидания по проверке и подтверждению качества

Если проблема демо на самом деле не в UI, а в недостающем игровом содержании, `prepare-demo` должен это явно показать и вернуть маршрут в `design-system` или `implement-system`.

## Что создает каждый шаг

| Шаг | Основной результат | Когда можно идти дальше |
|------|--------------------|-------------------------|
| `setup-engine` | `docs/technical-preferences.md` | стек, платформы и языковая политика определены достаточно, чтобы описывать системы |
| `map-systems` | `design/gdd/systems-index.md` | системы MVP и системы с высоким риском зафиксированы явно |
| `design-system` | один GDD для системы | одна система готова к реализации |
| `prototype` | `prototypes/[slug]/REPORT.md` | один риск закрыт, а выводы перенесены в канонические документы |
| `bootstrap-project` | минимальный рабочий каркас проекта | репозиторий реально запускается и структура вменяемая |
| `implement-system` | рабочий код для одной системы | система существует в коде и документы синхронизированы |
| `assemble-mvp` | `reports/mvp-assembly-report.md` | появился реальный игровой цикл |
| `playtest-and-tune` | `reports/playtest-report.md` | есть подтверждение текущего состояния, а не догадки |
| `prepare-demo` | `reports/demo-readiness.md` | критичные пробелы для демо и следующие шаги названы явно |

## Нормальная форма промта

Самая полезная форма запроса такая:

```text
Создай или обнови следующий правильный артефакт `gamedev` для этого проекта.
Идея игры: ...
Текущий этап: ...
Рабочий язык документации: ...
Язык для игрока: ...
Ограничения по платформе или стеку: ...
```

Если нужен ровно один шаг, говори его прямо и, когда полезно, называй сам навык:

```text
Используй `/design-system combat-loop`.
Напиши GDD боевой системы на русском языке.
Язык для игрока: английский.
Используй существующий индекс систем и не переходи к написанию кода.
```

## Правила, которые снимают большую часть путаницы

- Один проход `implement-system` должен реализовывать одну утвержденную систему, а не весь кластер фич.
- `prototype` это временное подтверждение гипотезы, а не рабочий код.
- `assemble-mvp` доказывает, что основной игровой цикл работает, а не скрытно производит пачку новых фич.
- `playtest-and-tune` это точечная настройка после реального запуска, а не угадывание.
- `prepare-demo` это контроль объема демо целиком, а не размытый запрос `сделай покрасивее`.
- Если документы начали смешивать языки, сначала исправь языковую политику в `game-concept` и `technical-preferences`, а потом синхронизируй остальные документы.
