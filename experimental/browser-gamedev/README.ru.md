# Браузерные игровые надстройки

Это дополнительные браузерные заметки для работы с веб-играми.

Они намеренно лежат вне основного пути `gamedev/`.
Они полезны, когда проект изначально браузерный и команде нужна дополнительная глубина по типу игры, браузерной отладке или подготовке оценочных промтов.

Эти файлы не являются отдельными скиллами.
Это справочные заметки, которые нужно использовать вместе с существующими скиллами репозитория, а не вместо них.

Используй эти файлы как дополнительные заметки:

- [`archetypes.ru.md`](archetypes.ru.md): короткая карта игровых типов и типовых рисков
- [`browser-debug-checklist.ru.md`](browser-debug-checklist.ru.md): практический список проверки для браузерной игры
- [`eval-prompt-pack.ru.md`](eval-prompt-pack.ru.md): нейтральные промты для ручных прогонов и будущих оценочных сценариев

Границы:

- `docs/technical-preferences.md`, `design/gdd/`, отчеты и маршрутизация по шагам должны оставаться в основном процессе `gamedev/`.
- для глубокой реализации под конкретный рантайм используй `Game Studio` или другого браузерного специалиста, если он доступен.
- относись к этой папке как к рабочим подсказкам, а не как к замене специализированной документации или документации движка.

## Как использовать вместе со скиллами

### 1. Когда выбираешь стек

Используй:

- основной скилл: `gamedev/setup-engine`
- дополнительную заметку: [`archetypes.ru.md`](archetypes.ru.md)

Зачем:

- `setup-engine` по-прежнему отвечает за `docs/technical-preferences.md`
- `archetypes.ru.md` помогает понять, какой именно тип браузерной игры вы строите

### 2. Когда описываешь системы и GDD

Используй:

- основные скиллы: `gamedev/map-systems`, `gamedev/design-system`
- дополнительную заметку: [`archetypes.ru.md`](archetypes.ru.md)

Зачем:

- заметка помогает точнее разложить браузерную игру на системы
- канонические артефакты все равно остаются в `design/gdd/`

### 3. Когда отлаживаешь браузерную игру

Используй:

- основные скиллы: `core/web-ui-smoke`, `core/web-ui-doctor`
- дополнительную заметку: [`browser-debug-checklist.ru.md`](browser-debug-checklist.ru.md)

Зачем:

- скиллы делают реальную работу в браузере
- список проверки помогает не гадать вслепую и не пропускать типовые проблемы браузерных игр

### 4. Когда готовишь оценочные сценарии

Используй:

- основной документ: `docs/gamedev-autoimprovement.md`
- дополнительную заметку: [`eval-prompt-pack.ru.md`](eval-prompt-pack.ru.md)

Зачем:

- основная схема оценок все равно живет в канонической документации
- набор промтов здесь только помогает быстро собрать заготовки для браузерных сценариев

## Пример маршрута

Типичный маршрут для браузерной игры:

1. Запусти `gamedev/setup-engine`, чтобы зафиксировать стек.
2. Если тип игры еще неясен, открой [`archetypes.ru.md`](archetypes.ru.md).
3. Потом используй `gamedev/map-systems`.
4. Затем используй `gamedev/design-system` для первой системы.
5. Для локальной браузерной проверки используй `core/web-ui-smoke`.
6. Если игра запускается, но ведет себя странно, подключай [`browser-debug-checklist.ru.md`](browser-debug-checklist.ru.md).

Правило переноса в основной контур:

- если одна из этих заметок показывает стабильную пользу на нескольких браузерных проектах, переноси устойчивые части в `docs/` или `gamedev/`
